# DAOs (Persistence Layer)

Inventario actual: **9 DAO interfaces** en `persistence-contracts` (`ar.edu.itba.paw.persistence`) + implementaciones en `persistence`. La mayoría de los DAOs principales (User, Product, Rent, Report, Block, Favorite, Price) ya están **migrados a JPA** (`*JpaDao` con `@PersistenceContext EntityManager`, JPQL + native queries para paginación 1+1, dirty checking para updates). Los legacy `*JdbcDao` que sobreviven usan `JdbcTemplate` + `SimpleJdbcInsert` y RowMappers `private static final`. **Cada tabla tiene un único DAO responsable** (regla cátedra-graded).

> Ver [[Hibernate & JPA]] para los patrones de mapeo, lazy/eager fetching y paginación 1+1 queries.

## Inventario

| DAO | Tabla(s) | Resumen |
|---|---|---|
| `UserJdbcDao` | `users`, `verification_tokens`, `password_reset_tokens` | CRUD usuario, tokens (issue/lookup/clear), updates granulares por field, búsqueda paginada para admin |
| `ProductJdbcDao` | `products` (+ JOINs) | CRUD, catálogo paginado con filtros (`CatalogCriteria` → SQL dinámico), `findOwnedBy`, `updateStatus`, búsqueda por título con `LIKE` (escapando `%`/`_`) |
| `ProductImagesJdbcDao` | `product_images` | Junction product↔image, `position` y delete por par (productId, imageId) |
| `ImageJdbcDao` | `images` | Insert binario, `findById` |
| `BlockJdbcDao` | `blocks` | Crear bloque, listar bloques activos por producto |
| `PriceJdbcDao` | `prices` | Default + especiales con rangos (`price_from`, `price_to`), overlap detection, `getCurrentPrice`, `getEffectivePricesForRange`, `findSpecialPricesForProduct` |
| `RentJdbcDao` | `rents` | CRUD, transición de status, queries para dashboards (renter / provider) con JOINs a producto y precio, paginación |
| `ReviewJdbcDao` | `reviews` | 1×rent, agregados (avg + count) por producto |
| `FavoriteJdbcDao` | `favorites` | Toggle (insert/delete), `existsByUserAndProduct`, listado paginado |

`AdminBootstrap` en services hace insert directo via `UserJdbcDao` (no es un DAO nuevo).

## Patrón

```java
@Repository
public class ProductJdbcDao implements ProductDao {

    private static final RowMapper<Product> PRODUCT_ROW_MAPPER = (rs, rowNum) -> ...;

    private final JdbcTemplate jdbcTemplate;
    private final SimpleJdbcInsert insert;

    @Autowired
    public ProductJdbcDao(final DataSource ds) {
        this.jdbcTemplate = new JdbcTemplate(ds);
        this.insert = new SimpleJdbcInsert(ds)
                .withTableName("products")
                .usingGeneratedKeyColumns("id");
    }
}
```

Convenciones:
- `enum.name()` para guardar; `valueOf` (o helper) para leer.
- `LocalDate`/`LocalDateTime` se convierten a `java.sql.Date`/`Timestamp` en el DAO; **fuera de `persistence` no aparece `java.sql.*`**.
- Cada tabla tiene **un único DAO** responsable.
- `RowMapper` reutilizable y `private static final` (ver `PRICE_ROW_MAPPER` compartido entre `PriceJdbcDao` y `RentJdbcDao` desde commit `99b59207`).
- Consultas paginadas devuelven `Page<T>` (`models.Page`) con `pageNumber`, `pageSize`, `totalCount`, `content`.

## Schema

### Tablas (PostgreSQL prod / HSQLDB tests)

`images`, `users`, `verification_tokens`, `password_reset_tokens`, `products`, `product_images`, `blocks`, `prices`, `rents`, `reviews`, `favorites`.

### Constraints destacadas

- `users.email` UNIQUE (lowercased).
- `users.role` CHECK ∈ `{USER, ADMIN}`.
- `products.status` CHECK ∈ `{ACTIVE, PAUSED, DELETED, PENDING_LOCATION, UNDER_REVIEW}`.
- `rents.payment_method` CHECK ∈ `{UPLOAD, CASH}` NOT NULL DEFAULT `UPLOAD` (V34, Issue #10).
- `users.admin_invite_token` UNIQUE (V35, Issue #20).
- `prices` con `price_from <= price_to` y separación default vs especial (default: `price_to IS NULL`).
- `rents.status` CHECK contra los 10 valores de `RentStatus`.
- `rents.payment_image_id` FK a `images` (nullable, comprobante de pago Sprint 2).
- `reviews` UNIQUE(`rent_id`) → 1 review por rent.
- `favorites` UNIQUE(`user_id`, `product_id`).

### Foreign keys
- CASCADE en `product_images`, `blocks.product_id`, `prices.product_id`, `rents.*`, `reviews.*`, `favorites.*`.
- SET NULL en `users.profile_picture_id` y `rents.payment_image_id`.

### Schema parity HSQLDB ↔ PostgreSQL

`persistence/src/test/resources/schema.sql` debe reflejar las migraciones de `persistence/src/main/resources/db/migration/` (Flyway). HSQLDB usa `INTEGER GENERATED BY DEFAULT AS IDENTITY`; PostgreSQL usa `SERIAL`. El test schema usa `sql.syntax_pgs=true` para tolerar la mayoría del SQL de prod. Ver [[Persistence (Spring JDBC)]].

`populator.sql` mantiene fixtures para read-tests (rule: nunca insertar desde el `Arrange` de un read test). Invariantes: cada producto tiene al menos un default price; clothing/footwear siempre con size; equipment/accessories nunca con size. Ver [[Testing Practices]].

## Migraciones recientes (Flyway)

- **V34** (`V34__add_rent_payment_method.sql`) — agrega `rents.payment_method VARCHAR NOT NULL DEFAULT 'UPLOAD'` con CHECK ∈ `{UPLOAD, CASH}`. Soporta el escape hatch CASH cuando el provider no tiene CBU (Issue #10). Ver [[Payment Method (UPLOAD/CASH)]].
- **V35** (`V35__admin_invite_token.sql`) — agrega `users.invited_as_admin BOOLEAN`, `users.admin_invite_token VARCHAR(128) UNIQUE`, `users.admin_invite_expires_at TIMESTAMP`. Habilita el flujo de invitación admin con TTL 7d single-use (Issue #20). Ver [[Admin Invite Flow]].

## Métodos nuevos de DAO

- `ProductDao.findByProvider(providerId, status, category, query, page, pageSize)` — variante filtrada para `MyListingsController` (excluye DELETED server-side).
- `ProductDao.updateStatusForLocation(location, fromStatus, toStatus)` — cascada activación al aprobar Location (PENDING_LOCATION → ACTIVE).
- `RentDao.updatePaymentMethod(rentId, paymentMethod)` — set del método de pago al initiate (Issue #10).
- `ReportDao.countByStatus(status)` — para badges del admin panel.
- `UserDao.createInvitedAdmin(email, token, expiresAt)` / `findByAdminInviteToken(token)` / `consumeAdminInvite(userId)` — flujo de invitación admin (Issue #20).
