# Persistence (Spring JDBC)

The project uses Spring JDBC (`JdbcTemplate` + `SimpleJdbcInsert`) for all database access. No ORM — every query is hand-written SQL.

> **Migración prevista a JPA/Hibernate.** Clase 7 introduce el reemplazo: ver [[Hibernate & JPA]]. Esta página describe el patrón actualmente en el código; convivirán hasta que se complete la migración.

---

## DAO Structure Rules

1. `RowMapper` as `private static final` constant — never instantiate per request
2. `@Autowired` constructor receives `DataSource`, creates `JdbcTemplate` and `SimpleJdbcInsert`
3. `SimpleJdbcInsert` with `.withTableName()` and `.usingGeneratedKeyColumns()`
4. Map keys in `create()` = column names
5. **NEVER create tables in Java code** — use `schema.sql` / Flyway migrations

---

## Example DAO Pattern

```java
@Repository
public class UserJdbcDao implements UserDao {
    private static final RowMapper<User> USER_ROW_MAPPER = (rs, rowNum) ->
        new User(rs.getLong("id"), rs.getString("name"), ...);

    private final JdbcTemplate jdbcTemplate;
    private final SimpleJdbcInsert jdbcInsert;

    @Autowired
    public UserJdbcDao(DataSource ds) {
        this.jdbcTemplate = new JdbcTemplate(ds);
        this.jdbcInsert = new SimpleJdbcInsert(ds)
            .withTableName("users")
            .usingGeneratedKeyColumns("id");
    }

    public User createUser(...) {
        Map<String, Object> args = new HashMap<>();
        args.put("name", name);
        args.put("email", email);
        Number key = jdbcInsert.executeAndReturnKey(args);
        return new User(key.longValue(), name, ...);
    }
}
```

## Enum Conversion

Enums are stored as strings using `.name()` and restored via `valueOf()` at the DAO boundary. Exception: `Size` uses display name matching via a `mapSize()` helper.

## Schema Management

Production schema managed via **Flyway** migrations at `persistence/src/main/resources/db/migration/`.

Current migration: `V1__initial_schema.sql`

### Two Separate Schema Files

- `db/migration/V1__initial_schema.sql` → PostgreSQL: `SERIAL PRIMARY KEY`, CHECK constraints
- `persistence/src/test/resources/schema.sql` → HSQLDB: `INTEGER GENERATED BY DEFAULT AS IDENTITY`

These are **incompatible** — that's why two files exist.

## DataSourceInitializer (for non-Flyway setups)

In simpler projects without Flyway, `DataSourceInitializer` can run a `schema.sql` at startup:

```java
@Value("classpath:schema.sql")
private Resource schemaSql;

@Bean
public DataSourceInitializer dataSourceInitializer(final DataSource ds) {
    final DataSourceInitializer dsi = new DataSourceInitializer();
    dsi.setDataSource(ds);
    dsi.setDatabasePopulator(new ResourceDatabasePopulator(schemaSql));
    return dsi;
}
```

The project uses Flyway instead — see [[Spring Configuration]] for the `flyway()` bean.

## HSQLDB vs PostgreSQL Schema Differences

HSQLDB and PostgreSQL are not fully compatible. Key difference for auto-increment primary keys:

| Database | Syntax |
|---|---|
| PostgreSQL (prod) | `userid SERIAL PRIMARY KEY` |
| HSQLDB (tests) | `userid INTEGER IDENTITY PRIMARY KEY` |

This is why two separate schema files exist. (source: clase-3-spring-jdbc-unit-testing.md)
